Repositories

In-memory storage that persists across HTTP requests and event handlers. Store, retrieve, and manage collections of data with simple actions.

What is a Repository?

A repository in ARO is an in-memory collection that persists for the lifetime of your application. Unlike regular variables that exist only within a single feature set execution, repositories maintain their data across multiple HTTP requests and event handlers.

Repositories are identified by names ending in -repository. This naming convention tells ARO to use persistent storage instead of ephemeral variable binding.

Key Characteristics

Persistent: Data survives across requests
Scoped: Isolated by business activity
Thread-safe: Safe for concurrent access
In-memory: Fast but not durable across restarts

Repository Operations

Store

Add data to a repository using the Store action:

(* Store a user object in the user-repository *)
<Store> the <user> into the <user-repository>.

(* Store a message *)
<Store> the <message> into the <message-repository>.

Each Store operation appends the value to the repository's collection. Repositories act like append-only lists by default.

Retrieve

Get data from a repository using the Retrieve action:

(* Retrieve all users from the repository *)
<Retrieve> the <users> from the <user-repository>.

(* Retrieve the last stored message *)
<Retrieve> the <message> from the <message-repository: last>.

(* Retrieve the first stored message *)
<Retrieve> the <message> from the <message-repository: first>.

Specifiers

Use specifiers to retrieve specific items from a repository:

SpecifierDescriptionExample
:last Get the most recently stored item <message-repository: last>
:first Get the oldest stored item <message-repository: first>
(none) Get all items as a list <user-repository>

Delete

Remove data from a repository:

(* Delete a specific user from the repository *)
<Delete> the <user> from the <user-repository>.

(* Clear all sessions for a user *)
<Delete> the <sessions> from the <session-repository>.

Complete Example

Here's a complete example showing a simple user service with repository storage: 

(* List all users *)
(listUsers: User API) {
    <Retrieve> the <users> from the <user-repository>.
    <Return> an <OK: status> with <users>.
}

(* Create a new user *)
(createUser: User API) {
    <Extract> the <user-data> from the <request: body>.
    <Create> the <user> with <user-data>.
    <Store> the <user> into the <user-repository>.
    <Return> a <Created: status> with <user>.
}

(* Delete a user *)
(deleteUser: User API) {
    <Extract> the <user-id> from the <pathParameters: id>.
    <Delete> the <user> from the <user-repository>.
    <Return> a <NoContent: status> for the <deletion>.
}

Chat Application Example

Repositories are great for real-time applications. Here's how to use them in a chat app: 

(* GET /status - Return the last posted message *)
(getStatus: Simple Chat API) {
    <Retrieve> the <message> from the <message-repository: last>.
    <Return> an <OK: status> with { message: <message> }.
}

(* POST /status - Store and broadcast a message *)
(postStatus: Simple Chat API) {
    <Extract> the <message> from the <body: message>.
    <Store> the <message> into the <message-repository>.
    <Broadcast> the <message> to the <socket-server>.
    <Return> a <Created: status> with { message: <message> }.
}

Repository Naming Convention

Any variable name ending in -repository is automatically treated as persistent storage: 

(* These are repositories (persistent) *)
<user-repository>
<message-repository>
<order-repository>
<session-repository>

(* These are regular variables (ephemeral) *)
<user>
<message>
<order>

Business Activity Scope

Repositories are scoped by business activity (the second part of a feature set name). This means different parts of your application can have separate repositories with the same name: 

(* This user-repository is scoped to "User API" *)
(listUsers: User API) {
    <Retrieve> the <users> from the <user-repository>.
    ...
}

(* This user-repository is scoped to "Admin API" - separate storage! *)
(listAdminUsers: Admin API) {
    <Retrieve> the <users> from the <user-repository>.
    ...
}

When to Use Repositories

Good for: Session data, caches, message queues, temporary collections, prototyping, and testing.

Not for: Production data that must survive restarts. Use a real database for persistent storage.

Events

Repository operations emit events that other feature sets can handle:

  • data.stored - Emitted when data is stored in a repository
  • data.deleted - Emitted when data is deleted from a repository
(* Handle user creation events *)
(Log User Created: DataStored Handler) {
    <Extract> the <repository> from the <event: repository>.
    <Log> the <log-entry> with "Data stored in " + <repository>.
    <Return> an <OK: status> for the <logging>.
}
